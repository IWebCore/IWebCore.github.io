    IMakeCore使用文档¶

    本篇文档描述 IMakeCore 的使用规范

    一个新的包管理系统¶

    在新兴编程语言领域，包管理 服务已是一个常态。java 有 maven， node.js 有 npm， python 能够 pip install 来安装三方库,go 语言有 go pkg,rust 有 cargo ,等等不一而足。

    在C++构建工具中，比较突出的是cmake，已经成为默认的事实标注。qmake 在GUI领域也有很重要的地位，但是 qmake 也被局限于GUI编程领域。xmake 作为国人开发的构建工具，颇受好评，市场接受度也越来越高。其他的也有Bazel 和 Meson等等。

    c++ 是一个古老而生机勃勃的语言，在对于库的支持上面因其起步较早，编译语言ABI不兼容的特性，在包管理机制上面落后于其他的语言。各家也提出了一系列的编译方案和包管理的方案，比如 vcpkg, conda, xmake 等等。这些方案各有千秋，发展迅速。其中 vcpkg 被广泛应用于软件开发。

    这些c++包管理工具的工作原理有的是将二进制的包封装起来给用户直接使用，有的是将源代码下载下来经过编译之后形成二进制库文件集成到软件接下来的编译中去，也有直接将库封装为 header-only 类型，通过inline的方式编译到程序当中。

    我在写 IWebCore的过程中，遇到了包管理的问题，就是我的 cpp 文件需要和程序文件一同编译，不能将我的代码预先编译成二进制库类型，也不能将代码封装成header-only的库类型给用户使用。原因很简单，我在库中 cpp 文件内使用了 静态全局变量在 main 函数执行前执行了一系列的代码，如果是 header-only的话，这部分代码将不会被执行，如果是库的话，对于静态变量/全局变量的处理也是一件麻烦的事情，所以我只能让我的库代码和程序代码一同编译。

    针对于上述问题，我封装了一个新的库管理程序，可以无缝的集成在 qmake和cmake编译系统的基础上。新的库管理程序相对于以前的库管理程序而言，可以将源代码封装后直接参与编译。当然，对于二进制库，和想编译成二进制库再集成进来，同样支持。

    优点不止如此，不管用户是在 qmake 上管理代码还是在cmake 上管理编译代码，该库管理都可以无缝集成，之后还会考虑更多的构建系统的集成，比如 xmake 等。用户只需要在构建文件上添加几行代码就可以使用该包管理系统，用户只需要在包配置文件中添加一个包的信息，该包就会自动添加到用户的构建当中。

    另外值得一提的是，一个包管理是工具是否能够火爆起来，一个是官方的支持，另外一个是民意的体现。在cpp领域，并没有一个官方的包管理系统被标准委员会所认同，而在民意中，大家是倾向于cmake和他的附属的包管理系统，这些包管理系统已经有了很多可用的包。对于IMakeCore而言，不是说，使用了vckpg的包，就不能在使用IMakeCore的包了，他们都能使用。使用IMakeCore并不影响用户使用vckpg等的包，而是IMakeCore给用户多提供了一个选择，一个更方便的选择。

    IMakeCore 会首先在 网络服务领域发挥他的价值。如果大家之后认可 IWebCore，喜欢使用 IWebCore编写自己的互联网程序，那么IMakeCore就会有更多的用武之地，大家也会更多的开发IMakeCore支持的分享到网络上，丰富IMakeCore的生态。而IMakeCore的包编写也是一个很简单的事情。

    一切就是这么简单！

    安装¶

    python 支持需要¶

    IMakeCore在底层实际执行功能的是 python代码，所以windows用户确保系统中有可用的python，python要求的版本是3.0 以上的版本。

    下载与安装¶

    我们先来安装这个包，在网络上下载该下载文件， IMakeCore.zip。解压该文件，如下所示：

    在这个文件中，有一系列的文件夹。.data 是配置文件存放的地方，.system 文件夹是功能集成的地方，其他的文件夹是IMakeCore预制的一些库，用户可以直接使用这些库，不用再去下载。

    在文件夹之外，有两个文件，windows_install.bat 是windows上执行安装的脚本，linux_install.sh 则是在 linux上执行安装的脚本，mac_install.sh 是在 mac 电脑上安装的脚本(关于mac_install.sh，只是先写在这里，我并没有mac电脑，也不知道对不对)。

    针对不同的平台用户执行不同的安装脚本，如下是用户执行 windows_install.bat 的脚本的示例：

    • 在windows 安装的过程中，由于脚本需要写入系统环境变量，他需要管理员权限才能安装，所以在执行脚本的时候，会弹出提示框提升权限。用户同意之后会执行接下来的操作。
    • 之后会弹出文件夹选择框，用户需要手动选择将要使用的文件目录，如果用户退出了文件选择，则会要求用户手动输入文件夹路径。
    • 在用户输入了安装文件夹之后，之后 IMakeCore会安装到该路径下，并且将该路径注册到系统环境变量中。这样用户就完成了安装。

    在 linux 系统中和 mac 系统中，安装过程是一致的，管理员权限，选择目录，然后安装。

    集成¶

    IMakeCore 可以通过在构建文件中添加一句话，就可以完成包管理系统的集成。

    Qt/QMake 集成¶

    添加方式¶

    QMake 集成需要在 项目的pro 文件中添加三行代码：

    1
2
3
    include($$(IQMakeCore))
IQMakeCoreInit()
include(.package.pri)

    第一行是将IMakeCore 需要的针对于QMake编译系统的包导入进来。

    第二行代码是 调用导入的包，在这个过程中，IMakeCore 会去寻找项目目录下的 package.json 文件，并解析该文件，如果没有这个文件，IMakeCore 会创建一个新的 package.json 文件。package.json 文件中定义了需要导入的包，之后IMakeCore 会尝试找到这些包，并将这些包的信息写入 项目目录的 .package.json 文件中去，就是我们第三行写的 include(.package.json) 的导入的新文件。

    在第三行中，将生成的.package.pri 文件导入进来，这样定义在package.json 中的包就会被集成进我们的系统中去了。

    如果开发者需要新的包导进来，需要修改 package.json，添加新的包即可。

    .cache 文件夹是存放一些缓存文件，这个文件夹可以被用户删除，但 IMakeCore不会主动删除该目录。

    packages.json 是我们在首次执行IMakeCore 时创建的文件，这个文件中包含库的声明，他的内容如下：

    CMake 集成¶

    cmake 集成和qmake 集成的内容基本一致，这里重复的部分不再记叙，讲一点不同的部分。

    添加方式¶

    CMake 集成只需要在 项目的CMakeLists.txt 文件里最后添加两行代码：

    1
2
    include($ENV{ICMakeCore})
ICmakeCoreInit(targetName)

    第一句话是在cmake 系统中导入 IMakeCore, 这个和 QMake 中集成 IMakeCore 的语句类似。不同的是 include 的参数不一致，这里是 $ENV{ICMakeCore}。$ENV{} 是cmake中引用环境变量的语法。 这句代码的意思是查找 ICmakeCore 的环境变量的值。所以include 的是 ICmakeCore环境变量的值 。而在 Qt中是 include($$(IQMakeCore)) ， $$() 是 qmake 中查找环境变量的语法，所以qmake中的意思是 导入 IQMakeCore 环境变量对应的值的文件。 注意这里 cmake 是 ICMakeCore， qmake 是 IQMakeCore， 两个的名称不一样。

    第二行代码 ICmakeCoreInit(targetName) 是初始化 ICmakeCore, 他是调用我们刚刚引入的文件里面的函数。 注意这里和 QMake 的 IQMakeCoreInit() 有区别的不仅仅是他的命名，更是 ICmakeCoreInit 里面传入了一个参数， targetName。

    在cmake构建中，是以 target 为中心进行构建的， 通过 add_executable来添加构建对象。这里的targetName 就是 executable 的构建名称。如果省略这个名称，则IMakeCore在执行的时候就会报错，告知开发者没有构建目标参数。 targetName 可以不仅仅只有一个，如果有多个 target， 可以传递多个targetName 进去，他们会共享同一套的包。

    另外需要注意的一个点就是，相对于QMake而言，cmake 少了一句话 include(.package.cmake)。这里并不是这句话没了，而是这句话被包含在了 ICmakeCoreInit 这个cmake 函数调用中去了。那为啥在cmake 中可以被包含，而在qmake 中没有被包含呢？ 这个是因为在qmake中如果在函数中包含这句话， qmake 管理系统在解析项目的时候会出现错乱。不能很好的识别我们要引入的库的内容。

    那能不能在cmake中填上这一句话呢，可以的，但是要在调用的函数中去掉这句话，然后手动写在 CMakeLists.txt 文件中。但是多一事不如少一事，多一行用户代码不如少一行用户代码。我就直接写在函数中了，不给用户添加额外的负担。

    项目 packages.json¶

    项目中 packages.json 用于管理 IMakeCore插件使用的文件。在上述示例中，我们只使用到了 packages.json 中的一个属性packages。在我写这篇文档的时候（今天是这个项目的第6天，开始写文档了，打算一个星期解决这个事情），我也一共定义和使用了三个属性，另外两个分别是 localized 和 libstore。

    packages.json 不像 node 工程里的 package.json 文件，来管理整个的项目。所以针对于项目上的package.json 我不准备定义很多的属性。（不定义，用户自己可以定义一些来玩， 比如 name， author 等，虽然也没啥意义）。在我们之后描述到的插件的 package.json 文档中，我们会定义很多，大量的属性。注意这两个描述文件，一个带s， 一个不带。

    除此之外，在后续的开发过程中，packages.json 里面会有一些和编译，和条件判断等相关的属性被添加进来。

    packages 列表¶

    packages属性的内容是一个列表，这个列表定义了我们要引入的库。下面讲述一下 包的属性。

    Package 包的属性¶

    group¶

    group 是组名，每一个库都会有一个组名，一个组里面可以有很多不同类型的库。IMakeCore 中默认的group 是 IWebCore。group 的名字应当只包含能够用于文件名称的，合法的ascii码，如中文字符等将被认为是非法的名称。

    name¶

    包的名字。名字应当只包含能够用于文件名称的，合法的ascii码，如中文字符将被认为是非法的名称。name 和 group 合在一起就是一个包在IMakeCore中的定位。

    version¶

    version 表示版本号。每一个包都有一个版本号，版本号 用 major.minor.patch 的方式标出。这里定义的版本号必须能够被 python 库 packaging.Version 识别。

    path¶

    path 是用户指定的库的路径。

    如果用户指定了 path 数据，IMakeCore在查找 包的时候将只能够在该路径下查找这个包。如果没有找到这个包，那么 IMakeCore 会报错，说找不到这个包。

    path的路径指向包的当前文件夹的路径，在这个路径中，必须存在package.json 文件和相应的导入文件。如果这两个文件不存在，则path指定的路径错误，包将不会被找到，IMakeCore会报错，找不到该包。

    url¶

    url 路径是用户指定的包下载路径。url 路径指向的内容必须是一个.zip 文件。文件里面包含 package.json 和 相应的导入文件。

    IMakeCore在解析该包的时候，如果没有找到该包，就会从该路径下载包，并将下载的包解压缓存，并将包导入到项目中去。如果下载出错，没有在 url 中找到该包，则 IMakeCore会报错，查找失败。

    repo¶

    repo 和 url 类似，是用于从网络上下载包的。具体的内容会在接下来项目的进一步开发中体现。

    localized¶

    localized是一个bool 类型的值，默认为false，表示这个包存在的位置。如果该值为 false，则不会强制缓存位置。如果该值为 true，表示该包的位置必须是在项目目录下的 .lib 文件夹内。

    在具体的查找过程中，如果发现该包不在 项目目录的 .lib 文件夹下面的时候，IMakeCore将会从缓存里面拷贝一份包到 该文件夹目录下面。

    这么做的目的是可以保证当前包的代码可以被用户修改，而不会影响全局。

    除了指定包的 localized 属性之外，packages.json 也有 localized 属性。

    单行定义方式¶

    引入库的定义可以很简单，也可以很复杂，简单的定义一行代码可以说清楚，复杂的情况需要指定更多的信息给 IMakeCore。

    单行定义 package 的完整结构如下：

    group/name@version¶

    这个是一个库的标准定义形式，需要有三方面的信息。group， name， 和version。这三个信息就可以确定出一个包。如下举例

    1
2
3
4
5
6
    {
    "packages":[
        "IWebCore/ICore@1.0.0",
        "asio/asio@2.0.0"
    ]
}

    单行定义的包查找的时候会首先查找各个缓存位置，如果缓存位置存在，则引用缓存位置。如果缓存位置不存在，则会从网络上尝试下载该包并引入。如果下载失败，则IMaekCore报错，找不到该包。

    除了上述的完整形式，单行定义 Package还支持如下的省略形式。

    group/name 类型¶

    在这种定义中省略了 version 的内容。 如果version 内容为空，是默认根据查找到的第一个缓存的位置。

    在上述的示例中，引入 json 库使用了一行的代码

    1
    nolhmann/json

    这个库是内置在 IMakeCore 中的，它不需要复杂的查询工作，就是简单的引入。

    nolhmann 是库作者的名字(他好像也不叫这个名字，和这个名字类似，但是他的github是这个名字)，这个名字在IMakeCore中，被称为 group (组)。在规划中，IMakeCore将来会有很多很多的库，那么总会有一些库的名字是冲突的， 所以通过组的概念来区分这些可能重名的库是一个很好的做法。json 则是库本身的 name (名字)。group 和 name 在一起标出一个库的身份。

    name@version 类型¶

    这里省略了 group 的内容。默认group 是 IWebCore。如下是示例：

    1
2
3
4
5
    {
    "packages":[
        "ICore@1.0.0",
    ]
}
    name 类型¶

    这种方式省略了 group 和 name， 这里group 是 IWebCore, version 是默认根据查找到的第一个缓存的位置。

    1
2
3
4
5
6
7
    {
    "pacakges":{
        "ICore",
        "ITcp",
        "IHttp"
    }
}

    扩展类的package 的定义¶

    包的声明可以是一个字串，也可以是一个对象类型，如下所示：

    {
    "packages":[
        {"name":"asio", "group":"asio"},
        {
            "name": "ITcp",
            "scope" : "local",
            "version": "1.0.0",
            "url":"http://127.0.0.1:8000/ITcp.zip"
        },
        {"name": "IHttp"},
    ]
}

    在上面的声明中，第二个包对象，使用了 url 的属性。关于属性的内容，这里不再多讲，参考上述文档。

    localized¶

    这个值默认是 false。如果用户将该值设置为 true， 那么所以的IMakeCore引入的库都将会从 项目目录下的.lib 文件就夹中引入。如果一个引入的包不在该目录中，那么这个包需要拷贝到该目录下，再行引入。

    将所以的包引入到目录下的目的是可以将该项目离线编译，并且可以随意移动。

    libstore¶

    libstore 是一个列表属性，这个属性里面存放一系列的路径。在IMakeCore 执行的时候，会从这些路径中读取所用的包。
